# mod_broadcast — Sprint Summary 2026-07-10

**Date range:** 2026-07-09 to 2026-07-10

## Executive summary

During this sprint, we successfully resolved race conditions during XML reload storms (W1), standardized exit reporting variables on the member channel (W4), added support for force-destruction with an optional grace period and announcement playback (W7), and implemented a complete DTMF caller-controls framework for both listener and speaker sessions (W-DTMF). All validation tests, including smoke tests, rapid reload storms, grace period timers, and the 12-action DTMF matrix, passed successfully. Deferred tasks include Valgrind diagnostics, a 30-minute soak, and verification against physical SIP handsets.

## Work items shipped

### W1 — Config reload race protection
* **Commit:** `747bde4edd`
* **Files touched:** `src/mod/applications/mod_broadcast/mod_broadcast.h`, `src/mod/applications/mod_broadcast/broadcast_config.c`, `src/mod/applications/mod_broadcast/broadcast_core.c`
* **Description:** Introduced global `config_rwlock` (rank 0.5 in the hierarchy) to synchronize concurrent configuration reloads with profile read operations. Profile lookups and cloning within `broadcast_create` are now wrapped in `config_rwlock` read-locks to avoid UAF or TOCTOU crashes during rapid reloads.
* **Mechanism:** The `broadcast_globals.config_rwlock` protects the global profiles list and caller controls list. Profile lookups (via `broadcast_config_find_profile`) require a read-lock (`rdlock`), which allows concurrent room creations to safely read the configuration. When a reload API command is dispatched, the reload thread acquires a write-lock (`wrlock`), blocking new creations, unloads the configuration pool safely, reloads the XML from disk, and releases the lock.

### W4 — Channel variables completeness
* **Commit:** `05e0e3feeb`
* **Files touched:** `src/mod/applications/mod_broadcast/mod_broadcast.c`
* **Description:** Added a dedicated `broadcast_member_set_exit_variables` helper to populate channel variables at teardown. The three new variables are:
  * `broadcast_ended_by_self` — set to `true` if the member left voluntarily (self-hangup or DTMF LEAVE), `false` if kicked by admin, wedge detector, or destroy.
  * `broadcast_kicked_by` — records who/what initiated removal (`admin`, `wedged`, `destroy`).
  * `broadcast_final_role` — records the final role of the member (`speaker` or `listener`) at the time of exit.
* **Mechanism:** Standardized session cleanup hook. When `broadcast_app_function` exits or is interrupted, `broadcast_member_set_exit_variables` computes the active duration using `switch_micro_time_now() - join_us` and formats it into milliseconds. If the member had speaker status (determined by `m->speaker_since_us > 0`), the duration spent as speaker is also computed. The `broadcast_left_reason` is extracted from the channel; if it is unset, it defaults based on active flags (`destroyed` or `kicked`).

### W7 — Destroy with grace period
* **Commit:** `90490a4bbf`
* **Files touched:** `src/mod/applications/mod_broadcast/broadcast_api.c`, `src/mod/applications/mod_broadcast/broadcast_core.c`, `src/mod/applications/mod_broadcast/broadcast_events.c`, `src/mod/applications/mod_broadcast/mod_broadcast.h`
* **Description:** Extended the destroy command with a new API signature: `broadcast <name> destroy [grace_ms] [announcement_file]`. The core function `broadcast_destroy_ex` clamps `grace_ms` to `60000` and plays the announcement file to existing listeners during the grace period while refusing new joiners via `BFLAG_DESTRUCT`. Emits the `broadcast::destroy-grace` event subclass.
* **Mechanism:** When a destroy command is issued with a grace period, `broadcast_destroy_ex` acquires a write-lock on the room's control lock, sets the `BFLAG_DESTRUCT` flag, and fires a `broadcast::destroy-grace` custom event on the FreeSWITCH event bus containing the delay interval. If an announcement file is specified, it sets the playback path. The speaker input thread is bypassed, and the playback file is played to all listeners concurrently via the SPMC ring buffer. Once the grace period expires, the room shuts down and hangs up all remaining sessions.

### W-DTMF — Full DTMF caller-controls framework
* **Commits:** `a5d03568c1` & `6f6313c10b`
* **Files touched:** `src/mod/applications/mod_broadcast/mod_broadcast.h`, `src/mod/applications/mod_broadcast/broadcast_config.c`, `src/mod/applications/mod_broadcast/broadcast_listener.c`, `src/mod/applications/mod_broadcast/broadcast_speaker.c`, `src/mod/applications/mod_broadcast/broadcast_events.c`, `src/mod/applications/mod_broadcast/conf/autoload_configs/broadcast.conf.xml`
* **Description:** Implemented 12 DTMF actions (9 new) using the XML `data` attribute for arguments parsed into `broadcast_control_binding_t`. Handled actions include Lua/JS execution, application executions, channel transfers, volume adjustments, and custom events. Added DTMF polling in the speaker loop and non-fatal load-time checks for `mod_lua` / `mod_v8`.
* **Mechanism:** The caller-controls module maps DTMF sequences (buffered inside `m->dtmf_buf`) to actions. The DTMF buffer is cleared after a 1.5-second timeout to prevent stray digit accumulation. When a digit sequence matches a binding, the action is dispatched. Actions include running JavaScript/V8 or Lua scripts via `switch_core_session_execute_application` (which checks if the loadable module exists first to log warning instead of crashing), running dialplan apps, transferring the channel, toggling muting flags atomically (`MFLAG_MUTED_SPEAKER`), and adjusting sln volume index granularly `[-12, 12]`.

### Key Concurrency Invariants Preserved

* **Speaker-Lifetime Invariant:** Foreign API threads (ESL, CLI, etc.) only write to the member's promotion signal. The speaker session thread itself is responsible for committing its state as `current_speaker` inside `broadcast_speaker_main_wait`, avoiding the TOCTOU crash.
* **housekeeping synchronization:** Housekeeping loops check for wedged consumers, checking `consumer_seq` age and calling the hangup routine under active thread safety.
* **Event Emitter Safety:** Reordered the module shutdown sequence so the emitter thread is joined *before* the broadcast memory pools are destroyed.
